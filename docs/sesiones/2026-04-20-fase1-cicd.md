## Debugging del workflow — 5 iteraciones

El workflow pasó por 5 commits antes de funcionar. Instructivo documentar cada fallo porque es aprendizaje para futuras ADRs/configs.

### v1 — `fc37811` — falló con "hostname contains invalid characters"

Causa: usé nombres de env vars `HOST` y `USER` en el step, que colisionan con variables de entorno estándar del runner Linux (`USER=runner` preseteado, `HOST` usualmente vacío). Mi `env:` block debería haberlas sobrescrito pero alguna quirk de GitHub Actions impidió la substitución.

Resultado: `$USER@$HOST` en el ssh target se expandió a `runner@` o `@ip` → ssh rechazó hostname inválido.

### v2 — `226091d` — renombré a VPS_USER/VPS_HOST — falló con "grep: Invalid regular expression"

El grep en Setup SSH tenía un `[` sin cerrar (character class incompleta): `grep -q "$VPS_HOST\|[" ~/.ssh/known_hosts`. Grep exit 2 → `if !` lo invirtió → triggered false error.

### v3 — `e7564d5` — arreglé grep con `[ -s file ]` check — falló con hostname inválido de nuevo

Setup SSH pasó ✓, pero Deploy step volvió a "hostname contains invalid characters" con `${VPS_USER}@${VPS_HOST}` expandiendo a algo malformado.

### v4 — `4a11a3c` — usé `${{ secrets.XXX }}` directo en lugar de env vars — mismo error

El display mostró solo `"***"` (GitHub masking) sin el resto del heredoc visible. Esto apuntó a que el string final quedaba malformado.

### v5 — `774d0ef` — strip defensivo con `tr -d '[:space:]'` — ✅ FUNCIONÓ

Root cause: **los secrets `VPS3_HOST` y/o `VPS3_USER` tenían un carácter whitespace al final** (probablemente un `\n` al pegar el valor con un Enter extra). Como las comillas dobles en bash preservan el whitespace, `"livskin@139.59.214.7\n"` se pasaba a ssh como argumento, y ssh rechazaba el hostname por contener caracteres inválidos.

Fix: strip whitespace al inicio del Deploy step:

```bash
TARGET_HOST=$(printf '%s' "$RAW_HOST" | tr -d '[:space:]')
TARGET_USER=$(printf '%s' "$RAW_USER" | tr -d '[:space:]')
```

Lección futura: **cuando un usuario paste un secret en GitHub UI, asumir que puede traer whitespace invisible**. Strip siempre antes de usar valores sensibles que fueron copy-pasteados.

## Lo que esto habilita

A partir de ahora, **cualquier cambio en `infra/docker/**` que hagamos por `git push` se despliega solo**. Beneficios:

1. **Velocidad:** 30-60s vs 5-10min manual de scp + ssh + restart
2. **Reproducibilidad:** el mismo workflow corre cada vez (sin "se me olvidó un paso")
3. **Auditoría:** GitHub Actions registra cada deploy con fecha, commit, quién lo disparó, qué archivos cambiaron, resultado
4. **Desde cualquier máquina:** desde la laptop de Dario en Milán o la del trabajo, mismo flujo
5. **Sin dependencia de Claude Code:** el workflow corre solo cuando hay push, no requiere que yo esté online
6. **Portfolio value:** demuestra patrón estándar de industria — deploy key + actions secrets + idempotent compose
