Nueva carpeta `infra/docker/brain-tools/`:

| Archivo | Propósito |
|---|---|
| `Dockerfile` | Python 3.11-slim + psycopg2-binary + requests |

Wrappers shell en `infra/scripts/`:
- `brain-index.sh`
- `brain-query.sh`

### Chunking strategy

Simple, char-based:
- `CHUNK_SIZE = 700` caracteres
- `CHUNK_OVERLAP = 150` caracteres  
- Corta preferentemente en `\n\n` > `. ` > `\n` para no romper oraciones

Trade-off aceptado: no es token-aware. En futuro (v2) se puede migrar a `MarkdownHeaderTextSplitter` para respetar headers.

### Idempotencia

Cada run del indexer:
1. Para cada archivo, `DELETE FROM project_knowledge WHERE source_path = X`
2. `INSERT` los nuevos chunks

Correr `index` muchas veces = seguro. Refleja estado actual del repo. Apto para cron semanal.

## Bug encontrado y arreglado

Primer intento falló con warning:
```
WARNING: The "BRAIN_USER_PASSWORD" variable is not set. Defaulting to a blank string.
```

**Causa:** `${BRAIN_USER_PASSWORD}` en `docker-compose.yml environment:` se interpola **a nivel YAML parse time**, leyendo de shell env (que está vacío). `env_file:` pasa vars al CONTAINER, no al parsing.

**Fix:** misma pattern que en alembic-brain/erp — construir `DATABASE_URL` desde env vars **dentro del código Python**, no en el compose:

```python
password = os.environ["BRAIN_USER_PASSWORD"]
DATABASE_URL = f"postgresql://brain_user:{password}@{host}:5432/{db}"
```

Lección confirmada: **secrets via env_file → consumir en aplicación, no interpolar en compose**.
